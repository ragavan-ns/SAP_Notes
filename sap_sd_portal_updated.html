ERP – Enterprise Resource Planning

Foundational concepts of ERP, its evolution and role in modern business.

🏭 What is ERP?

Enterprise = Organization / Company / Industry

Resource = Man, Material, Money, Machine/Equipment, Software, Assets etc.

Planning = Planning of all the above resources to run the enterprise smoothly.

📖 Definition: ERP is a business software system that is used to manage and streamline an organization's functions, processes and workflows with automation and integration. It is a business software that integrates with or connects all other business processes/modules/departments and shares a common database with one single application interface.

All the data is saved in a single database/server so it gives real-time data (Sales report/Procurement report). We can extract reports from this software so that we know how the business and its departments are working and help take good decisions.

⚠️ Problems Before ERP

Before ERPSystem Used
Sales and DistributionSQL
FinanceJava
Material ManagementMRP
ManufacturingDotnet

Issues before ERP:

  • No Tracking in process
  • Delay in delivery
  • Lost orders/customers due to delay
  • Data Duplications, Data inconsistency
  • Sharing of data between different systems is difficult
  • Lack of integrated information
  • Isolated decisions lead to overall inefficiencies
  • Increased expenses

✅ Advantages of ERP

  • You can track the business process — Manufacturing/Logistics/Finance
  • You can do the customize/enhancement in the system as per requirement
  • Tasks are automated to reduce human errors and cycle time
  • It gives accurate information for taking good decisions
  • Reduces the pressure and workload of manager

    • User Interface layer
    • Application layer
    • Database layer

SAP Landscape & Clients

System landscape arrangement, client types, transport requests and additional clients.

🌐 SAP System Landscape

System landscape is the arrangement of the various servers. A generic system landscape consists of a production system plus other systems such as Development and Quality systems.

  • Customizing or any changes are first done in the Development system and tested there locally
  • Then, changes are moved to the next level of testing in the Quality system or Testing system
  • Once the testing is complete, the changes can be moved into Production system — the change is said to have gone live

🖥️ Common Landscape (3 Servers)

1. Development / Golden ClientAll configuration/customization of client setting and new functionality are done here. Unit testing is done here with limited test data available.Only Consultants. No business user can access.
2. QualityTesting and verification of configuration functionality done here. No Configuration/Customization allowed. Only testing is allowed. SIT (System Integration Testing) and UAT (User Acceptance Testing) performed here.Consultants and Business Users
3. ProductionBusiness Live data. Only business user having the access to run the transactions. No testing allowed here.Business Users only. Consultants have display access only.

🔍 O2C Step Definitions (from Training Notes)

  • Inquiry: A sales inquiry is a non-binding (not legal commitment to buy) customer request for information like product availability or cost.
  • Quotation: A sales quotation is a formal, legally binding offer to a customer for goods or services.
  • Sales Order: A sales order is a formal agreement with a customer for products or services, containing details like prices, quantities, and delivery dates. Types: Standard Order, Return Order, Rush Order, Cash Sell etc.
  • Delivery: This document is to initiate the shipment process (Picking, Packing, Shipping, and Goods Issue / GI).
  • Billing / Invoice: It is the final stage of a sales transaction that generates an invoice/billing for a customer, which is then posted to Financial Accounting.
  • Payment: Is taken care of by Finance Team (FICO).
💡 O2C Cycle divided into 3 sections:
1. Pre Sales & Sales — Pre sales: Inquiry, Quotation | Sales: Sales Order
2. Shipping — Delivery
3. Billing — Billing / Invoice

📋 Pre-Sales Activities

  • Some marketing measures, such as direct mailing campaigns, Internet campaigns or trade fairs, can trigger the sales processes.
  • Possible sequences of such campaigns can be a non-binding customer inquiry or a request for a quotation.
  • Inquiries and quotations help you to determine important sales related data and can be saved as documents.
  • If the customer then places an order, this data can be further used in the system.

📦 Delivery Process Details

  • Shipping processing in Sales and Distribution begins when you create the delivery document.
  • The delivery document controls, supports and monitors various sub processes for shipping processing:
    • Picking and confirming (transfer orders)
    • Packing
    • Planning of transport (shipment document)
    • Posting the goods issue (goods issue documents)
  • Creating a delivery document includes copying information from the sales order such as the materials and quantities onto the delivery document.

🏛️ What is Enterprise Structure?

An organization structure is the framework of Roles and Responsibilities according to which the business is carried out in every Company or in a corporate.

For mapping Enterprise structures in SAP, different organizational units are available for: Sales, Shipping, and Billing.

  • We need to analyze the structures and procedures in the company and then match them to the SAP structures.
  • First we 'define' the various organization structures of the enterprise.
  • Then we 'assign' these structures logically to each other as per business requirements.
  • We should not use more organization structures than necessary to avoid complexity.
  • IMG nodes: Enterprise Structure → Logistics – General → Logistics Execution
💡 Example — TATA Group: TATA Automotive (Company Code), TATA Motors, TATA Steel, TATA Food, TATA Jewelry, TATA IT, Tata Electronics — all under one "TATA" brand/Company.

🏢 1. Company & Company Code

Company

  • It represents a Corporate Group or Group of Companies.
  • Business-independent entities.

Company Code (FICO — Head Quarters)

  • Smallest entity for external accounting.
  • Own balance sheet and profit and loss statement.
  • Examples: a company within a corporate group, or a subsidiary.
  • It is a legal entity where balance sheets are developed and P/L accounts are verified.
  • It is the place where all business/accounting transactions are recorded.
  • Field length: 4 digits (e.g. FI – 7333)
📌 IMG Path: SPRO → IMG → Enterprise Structure → Definition → Financial Accounting → Edit, Copy, Delete, Check Company Code → Execute → Edit Company Code Data (Double click) → Position

🏭 2. Sales Organization (SD)

  • It is the highest level of organizational unit in SD, responsible for the sale of certain products or services.
  • The responsibility of a sales organization may include legal liability for products and customer claims.
  • A Sales Organization is assigned to exactly one company code.
  • A Sales Organization is assigned to one or more Plants.
  • Each Sales Organization has its own master data — its own customer and material master data as well as condition records.
  • Field length: 4 digits (e.g. LG10)
📌 IMG Path: SPRO → IMG → Enterprise Structure → Definition → Sales and Distribution → Define, copy, delete, check sales organization → Execute → Define Sales Organization → copy from ZD10 to New SO

🔀 3. Distribution Channel (SD)

  • Distribution Channel is the means through which the goods reach the customers.
  • We can use different distribution channels to provide the market with the best possible service.
  • Distribution Channels provide a general structure for distributing goods. Wholesale trade, sales to industrial customers or direct sales from a plant are typical examples.
  • Customers can be served through one or more distribution channels within a sales organization.
  • Within a distribution channel, you can define your own master data for customers or materials as well as your own conditions and pricing.
  • Examples: Distributor, Retailer, Dealer, Direct Customer, Wholesaler
  • Field length: 2 digits (e.g. >9, >8)
📌 IMG Path: SPRO → IMG → Enterprise Structure → Definition → Sales and Distribution → Define, copy, delete, check distribution channel → Execute → Define distribution channel → New Entry → Save

📦 4. Division (SD)

  • A broad range of products the company is manufacturing is referred to as a division.
  • Material is always assigned to one division only.
  • A division can represent a certain product group.
  • Therefore, you can restrict price agreements with a customer to a certain division.
  • In addition, you can conduct statistical analysis by division.
  • Examples: Car Division, Truck Division
  • Field length: 2 digits (e.g. >9)
📌 IMG Path (Division): SPRO → IMG → Enterprise Structure → Definition → Logistics – General → Define, copy, delete, check division → Execute → Define Division (Double click) → New Entry and Save

🗺️ 5. Sales Area

A sales area is a combination of the sales organization, distribution channel, and division.

  • It defines the distribution channel a sales organization uses to sell products from a certain division.
  • Each sales and distribution document is assigned to exactly one sales area.
  • A sales area can belong to only one company code.
💡 Sales Area = Sales Organization + Distribution Channel + Division
Example: Sales Org 0001 → Dist Channel 01 → Product Division 01 = one Sales Area

🏪 6. Sales Office (SD — Optional)

  • A physical location (for example, a branch office) that has responsibility for the sale of certain products or services within a given geographical area.
  • Sales offices are assigned to sales areas.
  • The definition of sales offices is optional.
  • A selection field for the delivery and billing due list.
  • Field length: 4 digits (e.g. LG10)
📌 IMG Path: SPRO → IMG → Enterprise Structure → Definition → Sales and Distribution → Maintain Sales Office → Execute → New Entry and Save

👥 7. Sales Group (SD — Optional)

  • A group of sales people who are responsible for processing sales of certain products or services.
  • The definition of sales groups is optional.
  • A sales group is responsible for all items in a sales document.
  • You can assign a sales group to one or more sales offices.
  • You can assign people to a sales group.
  • Field length: 3 digits (e.g. LG1)
📌 IMG Path: SPRO → IMG → Enterprise Structure → Definition → Sales and Distribution → Maintain Sales Group → Execute → New Entry and Save

🏭 8. Plant (MM)

  • The plant is an operating area or branch within a company from which the goods are delivered to the customer.
  • It is also known as Delivering Plant from Sales view point.
  • Each plant is assigned to a unique company code.
  • Plant attributes:
    • A plant has an address.
    • A plant has a language.
    • A plant belongs to a country.
    • A plant has its own material master data.
  • Example: Flipkart / Amazon warehouse locations
📌 IMG Path: SPRO → IMG → Enterprise Structure → Definition → Logistics – General → Define, copy, delete, check plant → Execute

📦 9. Storage Location (MM)

  • A storage location in SAP represents the location in the plant where the goods are stored.
  • The stock is maintained at this level.
  • When the goods are received, the stock at the storage location goes up in the quantity received.
  • When the goods are sold, the stock at storage location is reduced according to the quantity delivered.
  • There may be one or more storage locations within a plant.
  • Stocks are managed only on a quantity basis and not on a value basis at storage location level.
  • Physical inventories are carried out at storage location level.
  • It is possible to store material data specific to a storage location.
  • Examples: FG01 (Finish SL), RM01 (Raw material SL)
📌 IMG Path: SPRO → IMG → Enterprise Structure → Definition → Materials Management → Maintain storage location (under Materials Management node)

🚚 10. Shipping Point (SD)

  • The physical location from which the items are shipped to the customer.
  • For example: a collection of loading ramps, bays, etc.
  • The shipping point is the top level in the organization for shipping.
  • The shipping points can be predefined, based on different combinations of:
    • Plants
    • Shipping Conditions
    • Loading Group
📌 IMG Path: SPRO → IMG → Enterprise Structure → Definition → Logistics Execution → Define, copy, delete, check shipping point → Execute. Also: Maintain loading point, Maintain transportation planning point

📋 What is Master Data?

Data necessary for processing business transactions can be stored in SAP as Master Data. Entering master data reduces business transaction processing time, as the system then proposes the relevant master data in the business transaction.

Master Data used in SAP Sales & Distribution:

  • 1. Customer Master Data (CMR)
  • 2. Material Master Data (MMR)
  • 3. Customer Material Info Record (CMIR)
  • 4. Pricing / Conditions Master
  • 5. Output Master

Master Data — Important Functions

  • Reduction of maintenance efforts
  • Control and consistency check of entered data
  • Avoidance of unnecessary redundancy
  • Enables use of data in different departments/modules
  • Ensures changes have to be maintained only once
  • Offers a solid basis for analysis and reporting
💡 The system automatically proposes master data (customer, material, price, text) while creating documents. This can be changed manually and copied further. Maintaining Master Records is crucial for optimum system performance.

🔬 Condition Technique in Pricing

  • In Sales and Distribution, the various elements used in the condition technique are set up in Customizing
  • During sales order processing, the system uses the condition technique to determine a variety of pricing information
  • For example, the system automatically determines which gross price the customer should be charged and which discounts and surcharges are relevant
  • Each of these pricing elements in a pricing procedure is called a condition type
  • The conditions can either be at header level (applicable to entire document) or at an item level
  • You can either enter the values for each condition type manually or determine them automatically
💡 Condition Technique is a backend setting done by the consultant to meet the pricing requirements of a business.

TOR & Availability Check

Transfer of Requirements (TOR), Material Planning, MRP, and Availability Check — from Accenture training slides.

🔄 What is TOR? (Transfer of Requirements)

  • When a sales order is raised, the system checks for availability of goods
  • If the required goods are not available, the system creates a Transfer of Requirement (TOR) for the supply of goods to PP (Production Planning)
  • PP can procure or produce the goods
  • This can be configured by creating requirement class and requirement type. The requirement needs to be checked in the corresponding schedule line category
  • The MRP department is informed about the quantities and deadlines by which incoming orders should be delivered
  • The system checks availability based on the requested delivery date of the customer and creates MRP records which contain all necessary information for passing on to planning

⚙️ Transfer of Requirements — Dependencies

  • It ensures that the goods are available in time for the delivery
  • Materials planning transfers the reported requirements and creates orders or purchase requisitions from them
  • The transfer of requirements is basically dependent upon the following factors:
    • Requirements type
    • Requirement class
    • Check group
    • Schedule line category

Prerequisites for TOR

  • Each requirement type must be allocated to one requirement class only
  • TOR must be switched on at requirements class level, the sales documents at schedule line level
  • We must define a check group. It is possible to have this check group proposed for the initial creation of a material master record
  • A plant must exist for transfer of requirements to be carried out at document item level
  • Requirements transferred to planning are further processed in MM module
  • We must, therefore, coordinate the transfer of requirements with MM consultant

🎛️ TOR Functionality

  • TOR is a prerequisite for carrying out the availability check. We can configure TOR which can work without an availability check but not the other way around
  • This is beneficial in scenarios where we don't want to execute the availability check for the material but want the requirement to be transferred to MRP for production/procurement decision making
  • There is no hard-and-fast rule which materials should be included for both availability check and the transfer of requirements, which should be included only for the transfer of requirements or which materials should be excluded from both
  • The decision depends upon the business requirements and the way MRP strategies are used to handle such requirements
  • Therefore, it is always a best practice to work closely with PP and MM counterparts while configuring the transfer of requirements and the availability check
  • A few common scenarios where the availability check is generally switched off are KANBAN relevant materials, bulk materials, slow-moving inventory materials and so on
  • In such cases, we control the procurement/production-related decisions using other planning strategies and methods available in the MM/PP application

🏭 Material Planning & MRP

Material Planning

  • The main objective of Material Planning is to guarantee material availability
  • It is required to procure or produce the required quantities on time for internal purpose or for Sales

MRP at Plant or Area Level

  • While running MRP at plant level, the system considers stocks from the storage locations within the plant, excluding the stocks already reserved
  • In case of MRP at area level, the stocks from the storage locations belonging to the same MRP area can be taken for planning to generate the requirements
  • Planning MRP at area level helps to restrict the planning to certain storage locations

MRP — Key Concepts

  • A schedule line in sales order contains information about material quantity ordered by the customer and the corresponding delivery date requested by the customer
  • The delivery date quoted by the customer is called the customer-requested delivery date
  • Material requirement is the delivery quantity requested by customer for the given date
  • Material requirement–related information is vital to the MRP department's ability to make production/procurement-related decisions
  • This information is transferred to MRP using the transfer of requirements (TOR)
  • We can transfer requirements to MRP either individually or collectively
  • An individual requirement transfers the material demand for each schedule line to MRP
  • A collective requirement transfers summarized data on a daily or weekly basis to MRP
  • We can trace back an individual requirement to the order from which it was originated, while the same is not possible in a collective requirement

✅ Availability Check (ATP)

  • Availability Check (AC) helps to confirm the schedule line in an order and thus facilitates the customer's order confirmation process
  • When carried out in the context of a sales order, the AC checks the stock availability for the material in the delivering plant
  • If sufficient quantity is available to meet the customer-requested delivery date, the AC then confirms the customer-requested delivery date on the schedule line
  • If requested delivery date cannot be met because of stock shortage, the AC is also capable of proposing a future date when the delivery quantities can be confirmed for an order

⏱️ Availability Check — Lead Times & Capacity

  • While calculating, AC considers various lead times such as:
    • Procurement / production lead time
    • Plant / warehouse processing time (including pick/pack, load/unload, transportation planning time, and goods issue/receipt time)
    • Shipment transit time
  • This helps to make accurate and reliable calculations about the delivery dates that we can promise to the customer
  • This also provides visibility for manufacturing/procurement capacity, warehouse processing capacity and transportation capacity

Availability Check — Prerequisites

  • Material availability date must be determined before ATP-check
  • Availability check must be activated in material master (view: sales general/plant)
  • If ATP-check is required in sales order: ATP-check has to be activated in Customizing for the schedule line category (requirement class)

↕️ Inward & Outward Movements

Inward Movement (Future Receipts)

  • Future receipts represent all the inward movement of goods that can add stock in the delivering plant/warehouse
  • Purchase order, production order and stock transfer order are a few examples of documents that trigger inward movement of stock

Outward Movement (Future Issues)

  • Future issues are all the outward movement of goods that can lead to the consumption of stock from the delivering plant/warehouse
  • Sales orders, deliveries, stock transfer orders, and assembly orders are a few examples of documents that trigger outward movement of stock
  • Production order, assembly order, and stock transfer order play dual roles — they are responsible for the addition as well as consumption of stock
  • This type of check is performed dynamically for each transaction
  • We can perform this check with or without replenishment lead time (RLT), which is the time to refill/replenish the inventory of the delivering plant

⚙️ How ATP Works — Step by Step

  1. The availability check process confirms the delivery date on the schedule line in an order. To confirm this date, the system uses the material availability date (MAD) as the basis
  2. This is the date on which the requested material quantity should be available to meet the customer requested delivery date
  3. The system determines the material availability date by backward calculation of the customer-requested delivery date
  4. It subtracts the delivering plant/warehouse processing time (pick/pack time, lead time for transportation planning, loading time) and transit time. This process is called backward scheduling
  5. The system then checks whether sufficient material quantity is available for the requested material in order to make the delivery as per the customer-requested delivery date
  6. If the requested quantity is not available, the system performs another search in the future, taking into consideration the planned receipts and issues

📅 What is Delivery Scheduling?

  • The goal of Delivery scheduling is to confirm a delivery date for a material requested by a customer
  • Delivery Scheduling is also known as Shipment Scheduling and has two steps: 1. Backward scheduling, 2. Forward scheduling
  • In shipment scheduling, we define the time at which the receipt of the goods at the customer can be acknowledged
  • Different lead times are taken into account: Preparation time, Loading time, Transportation lead time, Transit time

⬅️ Backward Scheduling

The system starts from the customer's Requested Delivery Date and works backward to determine the Material Availability Date.

💡 The system uses the longer of the two times (Loading time vs Transportation lead time) when scheduling deliveries. Note: The timeline runs right-to-left for backward scheduling — starting from delivery date and working back.

Backward Scheduling — Logic

  • For backwards scheduling, the customer's requested delivery date is used to calculate the material availability date and the transportation lead time
  • The outbound delivery must be created at the earliest of the two dates (selection date for the outbound delivery)
  • If both dates are after the order date and the material is available on the material availability date, the customer's requested delivery date can be confirmed
  • A schedule line is created for the sales document item
  • The date of the schedule line displays the confirmed delivery date (= the customer's requested delivery date)
  • If one of the two dates is before the order date, confirmation of the requested delivery date cannot take place — the system then attempts Forward Scheduling

Complaints Processing

Returns, Blocked Stock, Credit Memos, Delivery Free of Charge, Invoice Correction — from Accenture training slides.

🔄 Complaints Processing Overview

  • The handling of returns is also part of sales and distribution process
  • A quick and efficient complaints process assures important competitive advantages
  • If complaints are made regarding valuable goods, the goods have to be returned
  • The damaged material first has to be checked — it is stored in a special area for this purpose
  • If the complaint is justified, the customer receives a credit memo
  • It has to be ensured that the correct amount is credited to the customer account
  • Hence, the credit memo must be approved by an authorized manager before it is issued

Reasons for Returns

  • Faulty goods were delivered
  • The delivered goods were damaged
  • The customer ordered the wrong goods

After the goods are received by the plant: identification of the goods → clarification of the material number to find out when the goods were delivered.

📋 Returns Order Details

  • For the processing of returns, the order type RE (Returns) or RE2 (Advanced Returns) is used in standard SAP
  • Returns can be created with or without reference to a sales order or a billing document
  • Though the creation with reference is preferred (in that case, system receives all required data from the original document)
  • Item category for standard returns items is REN (Returns Item)
  • Returns processing requires that an approval step is carried out, which is implemented by an automatic billing block
  • This assures that no credit memo is created without the check if the complaint is legitimate
  • Only employee with special authorization is allowed to remove the billing block, usually Sales Manager
  • Entering of 'Order reason' is mandatory for later evaluation purposes in standard SAP

Returns Delivery

  • It is created with reference to the returns order
  • Standard returns delivery type: LR
  • Item category is adopted from the returns order
  • Item category REN is marked as "not relevant for picking" — so picking is not necessary in a returns delivery
  • After the goods arrived from the customer, the goods receipt has to be posted

Blocked Stock Returns

  • During goods receipt posting in the returns delivery, movement type 651 causes the posting of the stock into "Blocked stock returns" instead of "Unrestricted-use stock"
  • "Blocked stock returns" is non-valuated stock; therefore, no accounting document will be created for the concerned material document
  • Transfer to unrestricted-use stock: The transfer of goods from "Blocked stock returns" to "unrestricted-use stock" is done via transaction code MIGO_TR within SAP module MM
  • Transfer movement type: 453
  • "Unrestricted-use stock" is valuated stock

Defining mandatory fields, configuring procedures, status groups — from Accenture training slides and SAP system demonstrations.

📋 Introduction

  • An incompletion log defines how the system should respond when a sales document is incomplete
  • In other words, it guides the system behavior in the event that some of the necessary fields are not filled up
  • Typical choices include forcing the user to enter the data and not let the user save the order before all the necessary fields are filled
  • Incompletion procedure can be assigned to not just the sales order header level, but also line item level, schedule line level, delivery header and delivery line items
  • However, Incompletion log cannot be set for billing documents

  • Depending on the status group you can block certain subsequent activities for the document
  • For every field in the procedure you also have to define whether a warning message should be issued during processing if no data is entered in this field

🚚 Types of Shipment

  • One or more deliveries are involved in the shipment
  • One single point of departure and one single destination
  • In most cases, only one mode of transport is used

  • Several shipment documents are created

  • Special Shipment types are available for this

💥 How BOMs are Exploded

The BOM is exploded in the following order in the sales document:

StepWhat Happens
1Enter the material number that has a bill of material (main item). This is the main item of the BOM.
2The system determines the item category of the main item. This item category controls whether and how the BOM is exploded.
3If the BOM needs to be exploded, the system does it automatically. The components are listed as sub-items for the main item.
4The system determines an item category for each sub-item. The material and components in the BOM are controlled by item and schedule line categories.
💡 BOM Explosion Example — Desktop PC (Order 4098, Customer C100): Item 100: Desktop 1PC (main) → Item 110: Monitor 1PC → Item 120: Processor 1PC → Item 130: RAM 1PC → Item 140: Keyboard 1PC. The BOM is saved in SAP as a master record (CS01).

MODULE 5.6 · BILLING PLANS

Billing Plans

Billing plans allow you to store multiple billing dates in a sales document instead of a single billing date. Used for rental contracts, milestone projects, and service agreements.

📅 What is a Billing Plan?

  • When processing an order, instead of storing a single billing date, you can store a detailed billing plan with several billing dates in the sales document at item level.
  • You can also define a billing plan at header level, which is then valid for all items assigned to it.
  • These header billing plans are assigned to sales document types.
  • Depending on the business transaction you are processing, the system will automatically propose one of the two billing plan types.

🔀 Two Types of Billing Plans

TypeDescriptionUse CasesKey Characteristics
Periodic Billing
Type 02		 Bills the full amount periodically at regular intervals Rental contracts, Service contracts, Fixed period agreements, Total amount billing Bills same amount each period; start/end dates define duration; contract data drives the plan
Milestone Billing
Type 01		 Bills partial amounts or percentages at specific project milestones Plant construction & engineering, Project-based billing, Partial amounts, Final settlement Each milestone has its own billing date and value; used for long-term project contracts
💡 The system proposes the billing plan type automatically based on the sales document type configuration.

🔄 Periodic Billing — Details

  • Periodic billing is often used for rental and service contracts in order to bill the full amount periodically at certain dates.
  • The contract data stored in the system (installation data, end of contract) can then be used as the basis for creating the billing plan.
  • The start and end dates define the duration of the billing plan.
  • You can use the billing dates to determine when and how often billing is carried out.
  • For example, on the first day or the last day of every month.

MODULE 5.7 · REVENUE ACCOUNT DETERMINATION

Revenue Account Determination

Revenue account determination is the interface between the SD and FI modules. When a billing document is created, the system automatically posts to the relevant G/L accounts using the condition technique.

💡 Introduction to Revenue Account Determination

  • From the billing document, the system can post entries corresponding to different pricing elements viz. Price, Tax, Discount, etc., in the respective G/L accounts. This is revenue account determination.
  • Revenue account determination is the interface between SD and FI modules.
  • For automatic revenue account determination, the condition technique is used.
💡 G/L accounts determined include: Sales Revenue accounts, Sales deductions, G/L Accounts for Payment cards, Reconciliation Accounts. The determination posts automatically when a billing document is saved.

⚙️ Account Determination using Condition Technique

  • Account determination is carried out using the Condition technique.
  • Depending on the billing type, an account determination procedure is assigned for G/L accounts.
  • This determination procedure contains one or more condition types to which an access sequence is assigned.
  • This access sequence is composed of individual accesses in the form of condition tables.
  • The condition tables contain the fields and field combinations upon which revenue account determination depends.
  • The key fields are filled with document data, so that the system can search for a corresponding G/L account for the relevant combination.
  • If an access determines a G/L account, then it is set in the accounting document and the search is terminated.
📍 Condition Technique Building Blocks for Revenue Account Determination:
Condition Tables → Access Sequence → Condition Types (KOFI/KOFK) → Revenue Account Det. Procedure (KOFI00) → Assignment to Billing Type (e.g. F2)

🔧 System Conversion (Brownfield) Process

System Conversion converts an existing SAP ECC system to S/4HANA on the same system:

PhaseSteps
Discovery / Prepare PhaseSystem Requirements → Maintenance Planner → Pre-Checks → Custom Code Migration Worklist → Simplification List SAP S/4 HANA On-Premise
Realize PhaseSoftware Update Manager (SUM) with: Database Migration + Software Update + Data Conversion → Application-Specific Follow-On Activities
💡 The Simplification List is a key document listing all simplifications and changes in S/4HANA that may affect existing ECC functionality. Custom code must be checked against this list before conversion.

🌐 Selective Data Transition

  • Multiple source systems (SAP System A, SAP System B — any release, any DB) can be consolidated.
  • Core ERP components migrate to SAP S/4HANA (Core ERP applications and HANA DB).
  • Non-core or legacy components can remain in a Current / Legacy (Non-SAP) system.
  • Used when companies want to introduce new processes or consolidate multiple regional SAP systems into one global S/4HANA system.
